# 网页版 ↔ 小程序版双端同步机制

目的：每次网页版或小程序版发生修改时，都留下可复制给另一端会话的“同步包”，避免两边规则、素材、界面不同步。

## 一、固定原则

1. **先备份，再修改**：任何一端开始改动前，都要先备份被影响文件。
2. **先说明影响范围**：每次修改必须标明影响网页版、小程序版，还是双端。
3. **每次更新后生成同步包**：只要修改可能影响另一端，就要写一段“给另一端 Codex 的同步提示词”。
4. **不同端不要自动乱改**：用户明确说只改网页版，就只改 `web/`；但必须产出小程序同步提示词。用户明确说双端同步，才同时改 `web/` 和 `minigame/`。
5. **规则优先同步**：棋盘点位、起飞点、转直道入口、飞跃规则、骰子/任务卡触发规则，必须双端保持一致。
6. **素材变化要同步路径与压缩格式**：更换棋盘、图标、音效时，要同时说明素材文件名、目标路径、格式、是否需要压缩。

## 二、每次更新必须输出的“小程序同步包”格式

每次网页版修改完成后，在回复或记录中附上：

```text
【小程序同步包】
本次更新类型：新增 / 优化 / 修复 / 调整 / 素材 / 性能
网页版已改文件：
- web/...

小程序需要同步：是 / 否
如果需要，同步目标文件：
- minigame/game.js
- minigame/game.json
- minigame/assets/...

同步内容：
1. ...
2. ...
3. ...

关键规则/数值：
- ...

验证方法：
1. ...
2. ...

注意事项：
- 不要改 web/ 目录。
- 修改前必须备份 minigame/ 相关文件。
```

## 三、反向同步包

如果先改小程序，也要给网页版会话生成：

```text
【网页版同步包】
本次小程序已改内容：
- ...

网页版需要同步：是 / 否
同步目标文件：
- web/index.html
- web/assets/...

关键规则/素材路径：
- ...

验证方法：
- 打开 http://localhost:8080/web/index.html
```

## 四、双端规则表（当前版本）

### 机场位置

- 左上：黄色机场
- 右上：蓝色机场
- 右下：红色机场
- 左下：绿色机场

### 起飞点与转直道入口

| 颜色 | 起飞点 | 转直道入口 |
| --- | ---: | ---: |
| 红色 R | 23 | 7 |
| 黄色 Y | 49 | 20 |
| 蓝色 B | 10 | 33 |
| 绿色 G | 36 | 46 |

代码里如果使用 0 基索引，对应为：

```js
const START_INDEX = { R: 22, Y: 48, B: 9, G: 35 };
const ENTRY_INDEX = { R: 6, Y: 19, B: 32, G: 45 };
```

### 特殊飞行规则

- 黄色：14 → 26
- 蓝色：27 → 39
- 红色：40 → 52
- 绿色：1 → 13

触发条件：棋子刚好落到起飞格，才执行飞跃。

### 直道终点规则

- 进入直道后，必须正好点数到终点。
- 如果超过终点，超过的点数要反弹后退。

### 任务卡 / 国王卡规则

- 棋子移动完成后，再弹任务卡/国王卡。
- 骰子摇到 1：触发国王卡，同时仍应保留当前任务卡流程，不应让任务卡消失。
- 不是每个格子都触发国王卡，国王卡只由骰子点数 1 触发。

## 五、建议执行流程

### 只改网页版时

1. 备份 `web/` 相关文件。
2. 修改 `web/`。
3. 本地验证网页版。
4. 生成“小程序同步包”。
5. 如果用户确认小程序也要同步，再改 `minigame/`。

### 双端同步时

1. 备份 `web/` 和 `minigame/` 相关文件。
2. 先改网页版并验证。
3. 按同步包改小程序。
4. 检查两端规则表一致。
5. 更新 `CHANGELOG.md` 或同步记录。
6. 再提交/推送 GitHub。

## 六、建议给另一个 Codex 会话的开头提示词

复制下面这段给小程序会话：

```text
你是“飞行棋大冒险”小程序端同步会话。请只修改 minigame/ 目录，不要改 web/ 目录。每次修改前必须先备份相关文件到 backups/。请按照我给你的【小程序同步包】同步网页版的规则、素材或交互变化。同步后请说明改了哪些文件、如何验证、是否还有和网页版不一致的地方。
```
## 七、我 / Mavis / 小程序会话分工

参考写作 skill 的协作方式，把工作拆成“创造性决策”和“非创造力执行”。

### 1. 我负责的部分

适合由当前主会话处理：

- 和用户确认玩法、规则、界面方向。
- 判断棋盘点位、起飞点、入口、飞跃规则是否合理。
- 设计 UI 交互、动效节奏、任务卡/国王卡触发流程。
- 决定版本号、发布范围、是否需要双端同步。
- 生成给 Mavis 或小程序会话的明确执行包。
- 最终验收同步结果是否符合用户意图。

### 2. Mavis 适合承担的非创造力工作

适合分给 Mavis 的工作：

- 对照网页版和小程序版，检查规则常量是否一致。
- 把已经确认的规则从 `web/index.html` 搬到 `minigame/game.js`。
- 检查素材路径、文件名、大小写、格式是否一致。
- 检查是否漏了备份、CHANGELOG、同步包。
- 做重复性测试清单，例如：起飞点、入口、飞跃、骰子为 1、任务卡弹窗。
- 生成差异报告：哪些已同步、哪些未同步、哪些需要用户确认。
- 在不改变创意方向的前提下做小范围机械修复。

Mavis 不应该自行决定：

- 改玩法规则。
- 改棋盘点位含义。
- 改 UI 风格方向。
- 改用户已经确认的起点、入口、飞跃规则。
- 同时大改 `web/` 和 `minigame/` 而没有同步包。

### 3. 小程序会话负责的部分

小程序会话只负责 `minigame/`：

- 按同步包修改小游戏代码。
- 迁移确认过的规则、素材、音效。
- 在微信开发者工具中验证。
- 回报修改文件、测试结果、剩余差异。

## 八、给 Mavis 的任务包格式

当需要 Mavis 处理非创造力工作时，使用这个格式：

```text
【Mavis 执行包】
任务类型：规则同步检查 / 素材路径检查 / 双端差异报告 / 小程序迁移 / 发布前检查
禁止事项：不要改玩法设计，不要改用户已确认规则，不要改未指定目录。

背景：
- 项目：飞行棋大冒险
- 网页版目录：web/
- 小程序目录：minigame/

本次主会话已确认：
1. ...
2. ...

请 Mavis 执行：
1. ...
2. ...
3. ...

需要检查的文件：
- web/index.html
- minigame/game.js
- ...

输出要求：
1. 列出已完成项。
2. 列出发现的不一致。
3. 列出建议修改，但不要自行改变创意/玩法。
4. 如果改了文件，列出文件路径和验证方法。
```

## 九、推荐协作流程

### 只改网页，但要提醒小程序

1. 主会话确认用户需求。
2. 主会话备份并修改 `web/`。
3. 主会话生成“小程序同步包”。
4. 如果内容复杂，再生成“Mavis 执行包”，让 Mavis 先做双端差异检查。
5. 用户把同步包交给小程序会话，或确认由当前会话继续改 `minigame/`。

### 双端都要改

1. 主会话确认规则和界面。
2. 主会话先生成明确同步包。
3. Mavis 做机械检查：规则常量、素材路径、文件差异。
4. 小程序会话按同步包改 `minigame/`。
5. 主会话做最终验收和 GitHub 同步。

### 发布前

1. Mavis 检查备份是否存在。
2. Mavis 检查 `web/` 与 `minigame/` 规则是否一致。
3. 主会话确认是否提交 GitHub。
4. 推送后给用户线上地址和缓存刷新方法。
