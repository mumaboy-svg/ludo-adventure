# 飞行棋大冒险更新流程

这份流程用于每次修改游戏前后的版本管理、备份、测试、说明和发布。目标是让每个版本都能追踪、能回退、能对外说明。

## 一、版本号规则

采用游戏项目常见的三段式版本号：`主版本.功能版本.修订版本`，例如 `v1.2.3`。

强制规则：每轮修改开始前评估改动量，完成后根据实际差异复核版本号。界面中不得长期固定显示旧版本号，也不得因为尚未发布就一直沿用同一个版本号。

### 0. 每轮版本评估表

每次更新先回答以下问题：

1. 是否改变核心玩法、棋盘结构、存档兼容性或整体操作流程？是则升级主版本。
2. 是否新增玩家可感知的功能、页面、完整 UI 模块或双端能力？是则升级功能版本。
3. 是否仅修复错误、替换素材、微调样式或优化性能？是则升级修订版本。
4. 同一轮同时包含多个级别时，采用其中最高级别，不重复累加。

改动量参考：

| 改动量 | 版本变化 | 示例 |
| --- | --- | --- |
| 小 | `2.1.0 → 2.1.1` | 修复点数显示、文字溢出、路径或轻量性能问题 |
| 中 | `2.1.1 → 2.2.0` | 重做任务中心、增加结算页、完成一组大厅与 HUD 功能 |
| 大 | `2.2.0 → 3.0.0` | 核心玩法重做、存档不兼容、整体流程重构 |

开发中尚未正式发布的零散修改先记入 `CHANGELOG.md` 的“未发布”。当一组内容达到可验证、可交付状态时，再确定并展示新的正式版本号。

### 1. 主版本：重大升级

格式：`v2.0.0`、`v3.0.0`

适用情况：

- 游戏规则大改，旧玩法和新玩法明显不同。
- 棋盘结构、胜利条件、任务系统等核心机制重做。
- 存档格式不兼容旧版本，旧存档可能不能继续使用。
- UI 和交互整体重构，玩家需要重新适应。

示例：

- 从普通飞行棋升级为“飞行棋 + 大冒险剧情模式”。
- 新增完整关卡系统、角色成长系统，并改变原来的游戏流程。

### 2. 功能版本：中等内容更新

格式：`v1.1.0`、`v1.2.0`

适用情况：

- 新增一组明显功能，但不破坏旧玩法。
- 新增任务库、音效开关、玩家头像、胜负结算页、设置项。
- 明显优化手机端或电脑端布局。
- 新增可见的新内容，例如新棋子、新任务分类、新主题。

示例：

- `v1.1.0`：新增任务库导入导出和示例任务。
- `v1.2.0`：新增移动端适配和胜利结算界面。

### 3. 修订版本：小修小补

格式：`v1.0.1`、`v1.0.2`

适用情况：

- 修复 bug。
- 修正文案、路径、音效引用、图片显示问题。
- 小范围样式调整。
- 不改变玩法，也不新增大功能。

示例：

- `v1.0.1`：修复骰子音效路径。
- `v1.0.2`：修复长名字遮挡棋子。

## 二、更新类型分类

每次更新说明里建议按以下分类记录，方便以后回看。

- 新增：新功能、新内容、新素材。
- 优化：体验、布局、动画、性能、操作流程改善。
- 修复：bug、路径错误、显示异常、存档异常。
- 调整：规则数值、按钮位置、文案、默认配置变化。
- 文档：README、更新说明、测试记录等项目文档变化。
- 已知问题：本次还没有解决，但需要记录的问题。

## 三、每次更新的标准流程

### 1. 更新前准备

1. 确认当前版本号和要改的内容范围，填写本轮版本评估结果。
2. 查看 `git status`，确认当前是否有未提交改动。
3. 每次开始修改前必须先做备份，确认可以回退后再动文件；不能跳过这一步。
4. 如果当前版本已经稳定，先打备份标签，例如 `v1.0.0`。
5. 如果是较大更新，新建更新分支，例如 `codex/v1.1.0-mobile-layout`。
6. 每次网页端更新完成后，都要写一条“小程序同步说明”，说明小程序端是否需要跟进，避免两边功能不同步。
7. 涉及双端规则、素材、棋盘、音效、任务卡、移动动画时，必须按 `docs/sync/双端同步机制.md` 生成同步包。

### 2. 备份规则

备份是每次更新的必做步骤，不论是小修复、素材替换、规则调整还是大版本升级，都要先保留可回退版本。

强制规则：没有备份点，不开始修改。出现问题时，优先退回 Git 里的稳定提交，不在问题代码上反复叠加修补。

优先使用 Git 作为正式备份方式：

- 更新前：先确认当前改动是否需要提交、打 tag 或复制到 `backups/`。
- 改动前备份点：如果当前工作区是稳定状态，先提交或记录当前 commit hash，作为本轮修改的回退点。
- 稳定版本：使用 Git tag 标记，例如 `v1.0.0`。
- 开发过程：每完成一组独立改动就提交一次 commit。
- 重要更新：推送到 GitHub，保证远程也有备份。

最低要求：每次更新前至少保留一种备份方式，可以是 Git commit、Git tag、GitHub 远程分支，或 `backups/` 文件夹中的手动备份。

需要额外保留文件备份时，放到 `backups/`，命名格式：

```text
backups/YYYY-MM-DD_v版本号_说明/
```

示例：

```text
backups/2026-06-29_v1.0.0_original-import/
```

注意：`backups/` 只用于特别重要的手动备份。日常版本回退以 Git tag 和 commit 为准。

### 2.1 回退规则

当某个模块连续修复后仍然不稳定，例如骰子动画、棋盘坐标、任务弹窗等，应停止继续叠加补丁，改用 Git 回退策略：

1. 先记录当前问题和截图。
2. 查看稳定提交：`git log --oneline`。
3. 优先从 Git 里的稳定提交恢复对应文件或对应模块。
4. 只恢复出问题的模块，不随意回退整个项目。
5. 恢复后重新做小范围修改，并再次记录“小程序同步说明”。

当前可参考的稳定提交点：

- `d25afe9`：游戏中界面美化基础版，可作为网页 UI 美化前后的重要回退点。
- `19fb444`：UI 图片转换为 WebP 后的版本。
- `9eafecf`：棋盘点位与骰子显示校准相关版本。
- `637446e`：颜色起点和入口点修复后的版本。

如果骰子、棋盘、任务卡等局部功能修坏，优先从这些 Git 提交中提取对应模块恢复，而不是继续在坏版本上反复改。

### 3. 修改实现

1. 小改动直接修改对应文件。
2. 中大型改动先写清楚目标，再分阶段修改。
3. 网页版正式入口为 `web/index.html`；根目录 `index.html` 是 GitHub Pages 兼容跳转页。
4. 网页版图片放 `web/assets/images/`，音效放 `web/assets/audio/`，测试页放 `web/tests/`；小游戏素材放 `minigame/assets/`；文档放 `docs/`。

### 4. 测试检查

每次发布前至少检查：

- 首页能打开。
- 图片素材能显示。
- 骰子音效能加载，不能播放时有降级方案。
- 开始游戏、玩家设置、掷骰、移动、任务弹窗可用。
- 本地存档、继续上局、清空存档可用。
- 手机宽度和电脑宽度布局不明显溢出。
- 浏览器控制台没有影响游戏运行的报错。

详细检查项见 `docs/release/发布检查表.md`，双端发布步骤见 `docs/release/发布说明.md`。

双端同步检查：如果本次改动影响另一端，发布前必须检查是否已生成“小程序同步包”或“网页版同步包”。同步包模板见 `templates/小程序同步包模板.md`。

### 5. 更新说明

每个正式版本都要更新：

- `CHANGELOG.md`：写给维护者看的完整变更记录。
- `docs/updates/YYYY-MM-DD_v版本号_更新主题.md`：每轮修改都必须新建一份独立说明，不能只写在聊天记录里。
- 更新公告：写给玩家看的简短说明，可从 `templates/更新公告模板.md` 复制。
- README：只有当目录、打开方式、发布方式变化时才需要改。
- 网页端和小程序端的可见版本号必须保持一致；发布前检查页面标题、大厅角标、`APP_VERSION` 和 `GAME_VERSION`。

独立说明文件至少包含：版本评估、修改文件、具体改动、明确未修改的高风险模块、测试结果、双端同步状态、备份路径和发布状态。即使本轮只修一个显示问题，也必须保留这份记录。

事务性检查和测试分工按 `docs/协作分工与测试准入.md` 执行。Mavis、QClaw 可以产出清单和测试报告，但最终版本号、正式代码验收、Git 提交和发布由 Codex 负责。

### 6. 发布与上线

正式发布前先确认发布方向：网页版、微信小游戏版，或两者同时发布。详细发布入口和双端步骤见 `docs/release/发布说明.md`。

推荐 Git 顺序：

1. `git status`
2. `git add ...`
3. `git commit -m "release: v版本号 更新标题"`
4. `git tag v版本号`
5. `git push origin 当前分支`
6. `git push origin v版本号`
7. 如果使用 GitHub Pages，打开线上地址检查一次。
8. 如果发布微信小游戏版，在微信开发者工具中完成上传前检查。

## 四、版本命名建议

版本标题建议使用“版本号 + 简短主题”。

示例：

- `v1.0.0 项目整理版`
- `v1.1.0 移动端体验优化`
- `v1.2.0 任务库升级`
- `v1.2.1 修复音效路径`

## 五、发布节奏建议

- 小修复：可以当天发布，版本号加修订号。
- 一组体验优化：合并成一个功能版本发布。
- 大玩法变化：单独开主版本，不和小修复混在一起。
- 没测试完的功能：不要进正式版本，可以留在开发分支。

## 六、视觉美化推进

后续 UI/视觉升级按 `docs/视觉美化规划.md` 执行。该文档把美化任务分为：

- 简单快速完成：优先做，不依赖新图片，风险低。
- 需要图片支持：先准备素材，再转 WebP 并落地。
- 大动作更新：放后面，单独备份、单独测试、必要时单独提交。

每轮视觉更新仍然必须遵守本文的备份、测试、更新说明和“小程序同步说明”规则。

## 七、当前版本基线

当前双端开发基线为：

```text
v2.2.0 双端体验与小程序功能升级
```

后续从 `v2.2.0` 继续按本规则递增，不再使用笼统的 `v2.0`。
