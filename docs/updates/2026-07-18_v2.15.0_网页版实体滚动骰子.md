## 版本评估

本轮将网页版骰子从基础 CSS 3D 翻转升级为完整的抛出、滚动、弹跳和落地表现，属于玩家可明显感知的完整动效能力，因此网页版功能版本从 `v2.14.0` 升级为 `v2.15.0`。

本轮是 Web-only 更新。微信小游戏没有修改，继续保持 `v2.14.0`。

## 实际修改

### 固定六面

- 骰子六面固定为正面 1、背面 6、顶面 2、底面 5、右面 3、左面 4。
- 动画期间不再替换每个面的点数内容，只旋转同一个实体方块。
- 1–6 分别使用稳定的最终旋转映射。
- 骰子整体使用 ARIA 点数标签，最终点数与视觉结果一致。

### 实体滚动表现

- 使用 `requestAnimationFrame` 驱动连续动画。
- 模拟横向抛出、第一次落地、第二次小弹跳和最终落定。
- 阴影随高度缩放、变淡和模糊。
- 两次碰撞和最终落地会触发短促光圈。
- 根据骰子台宽度使用桌面、普通手机和小屏手机三档运动幅度。

本实现是经过设计的运动曲线模拟，不是物理引擎。

### 移动端修复

- 骰子面类名改为 `face-top`、`face-bottom` 等独立名称。
- 避免项目已有移动端 `.top { display: none; }` 误伤骰子顶面，导致最终点数 2 空白。
- 竖屏骰子操作区改为骰子在上、按钮在下的居中布局。
- 收紧竖屏 3D 舞台尺寸并保留旋转安全边距，修复骰子偏右和边缘被裁切。
- 竖屏专用样式仅将 `.dice-container` 改为 `overflow: visible`，解除 3D 骰子在极端旋转角度下被圆台裁切的问题。
- QClaw 短审确认现有 `perspective`、`preserve-3d` 和内部层级链正确，因此没有追加多余的 `z-index` 或 `isolation`。
- 高度不超过 `720px` 的矮竖屏使用紧凑尺寸，减少对底部操作区的挤压；横屏布局保持不变。

### 结果契约保持不变

- 每次掷骰仍只创建一个冻结 transaction。
- 随机结果仍只生成一次。
- 动画、`data-final-roll`、ARIA、“上次点数”、任务回调和玩法结算继续共用同一点数。
- 动画期间继续使用原有连点锁。
- 继续使用现有 `dice_roll.mp3` 和原有音效降级路径。

## 验证结果

- 网页专项测试：`38/38` PASS。
- 固定六面：1/6/2/5/3/4 映射 PASS。
- 最终旋转：1–6 PASS，2 点与 5 点独立检查 PASS。
- 30 次确定性 transaction：随机源每次只调用一次 PASS。
- 真实随机抽查：HUD、`data-final-roll` 与 ARIA 点数一致。
- 视口：`1366×768`、`844×390`、`390×844`、`375×667` 的骰子区域无横向裁切。
- 竖屏额外检查 `390×844`、`375×667`、`360×640` 的骰子台居中、旋转边界和按钮排列。
- 竖屏骰子改为独立占据最上层并水平居中，掷骰按钮位于骰子下方居中；`360×640` 紧凑模式进一步压缩 8px，避免底部工具区掉出第一屏。
- `360×640` 动态掷骰实测中，旋转骰子完整可见；落定结果为 4 时，HUD、`data-final-roll` 与 ARIA 三处均为 4。
- 游戏预览页控制台无错误或警告；测试页 `38/38` PASS。
- `375×667` 整页仍有既有纵向滚动，未发现由本轮骰子造成的新横向溢出。
- 网页内联脚本和差异格式检查通过。
