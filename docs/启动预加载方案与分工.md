# 微信小游戏启动预加载方案与分工

- 记录日期：2026-07-17
- 计划版本：`v2.10.0`
- 实施端：`minigame/` 启动预加载 + Web/小游戏横屏大厅动态补齐
- 当前状态：双端真实加载流程与正式视觉已接入；网页版多视口验收和小游戏开发者工具预览已完成，等待微信真机最终验收

## 目标

为微信小游戏增加真实资源预加载流程。启动后先显示品牌加载页，根据资源实际加载结果更新进度，关键资源准备完成后淡入游戏大厅。同时补齐网页版和小游戏横屏大厅动态，让横屏与手机版使用同一套轻量视觉语言。

加载页同时承担资源错误诊断：记录加载失败的文件路径，区分路径错误、文件缺失和真机格式兼容问题，避免资源不显示但没有明确提示。

## 启动流程

1. 创建 Canvas 后进入 `loading` 场景，不立即进入大厅。
2. 显示轻量背景、游戏 Logo、飞行动画、真实进度条、百分比和版本号。
3. 第一阶段加载大厅关键资源：大厅背景、Logo、三个主按钮、任务中心/设置/棋子预览图标和大厅飞机。
4. 第一阶段成功后允许进入大厅。
5. 第二阶段继续加载棋盘、四色棋子、任务卡、国王卡、骰子音效和游戏界面资源。
6. 背景音乐只创建播放器，不阻塞进入大厅；仍遵守微信首次用户操作和音乐开关规则。
7. 关键资源加载失败时显示失败状态和“重新加载”按钮；非关键资源失败时进入大厅并在后台重试。

## 进度规则

- 进度必须由已完成资源数和资源总数计算，不使用纯计时假进度。
- 每项资源只结算一次，成功和失败都必须进入完成计数，避免进度卡死。
- 关键资源全部成功前不得自动进入大厅。
- 加载完成后使用短暂淡出/淡入过渡，避免画面突然切换。
- 加载超过 8 秒时显示重试入口和失败资源摘要。

## 视觉方向

- 保持《冒险小飞行家》现有明亮、糖果岛和四色飞行棋风格。
- 中央使用大 Logo；Logo 下方是一架沿进度条前进的小飞机。
- 进度条可使用红、黄、蓝、绿四色连续填充。
- 文案保持简洁：`正在准备冒险 65%`。
- 底部显示当前版本号。
- 不增加“点击开始”；关键资源加载完成后自动进入大厅。
- 所有内容必须适配手机首屏、安全区和不同纵横比，不得拉伸 Logo 或按钮。

## 资源兼容约束

- 棋盘继续使用已验证兼容的 JPG。
- Logo、任务卡和国王卡在真机中无法显示透明 WebP，必须使用压缩 PNG。
- 透明关键图片不能仅凭桌面工具可解码就判定通过，必须完成微信真机预览。
- 新增资源优先控制尺寸和包体；是否使用 WebP 必须按真机验证结果决定，不再统一强制转换。
- 小游戏主包上传大小必须低于 4MB。

## 会话分工

### 当前主开发会话

负责以下工作，并且是唯一允许修改 `minigame/game.js`、版本号和发布文件的会话：

1. 修复 Logo、任务卡和国王卡的 PNG 真机兼容问题。
2. 实现真实预加载状态机、进度统计、错误收集、重试和场景过渡。
3. 接入美化会话交付的最终素材和布局参数。
4. 处理资源包体、备份、版本号、CHANGELOG 和更新说明。
5. 执行语法检查、资源路径检查、微信开发者工具预览和上传。

### 整体美化会话

只负责视觉设计，不直接修改正式游戏代码。交付内容：

1. 加载页背景方案或最终轻量素材。
2. Logo 的位置、最大尺寸和安全区建议。
3. 飞机动画样式、运动轨迹和循环时长。
4. 进度条颜色、尺寸、圆角、文字样式和间距参数。
5. 至少提供 375×812、390×844、430×932 三种手机尺寸的布局检查结果。
6. 列出素材文件名、像素尺寸、格式、透明通道和压缩后大小。

本轮由整体美化会话（Codex）具体负责：

1. 先复用现有 `title_logo.png`、`home_plane_flyby.png` 和大厅色彩体系，判断是否确实需要新增图片；没有必要时不要求用户重复生成素材。
2. 输出一份可直接实现的加载页布局规格，包含安全区、坐标比例、Logo 最大宽度、进度条宽高、文字字号和飞机轨迹。
3. 为 `375×812`、`390×844`、`430×932` 三档竖屏提供尺寸检查表，确认 Logo、进度条、百分比、版本号和重试提示均在首屏内。
4. 定义加载中、加载成功、关键资源失败和非关键资源失败四种视觉状态，但不改资源加载逻辑。
5. 检查主开发会话完成后的截图或真机结果，重点验收拉伸、遮挡、透明 PNG、文字溢出和安全区；发现问题时只提交参数修正建议，由主开发会话落入 `minigame/game.js`。
6. 维护本文件中的视觉交付状态，避免主游戏 HUD 与预加载页同时占用 `v2.10.0`。
7. 在 `web/index.html` 补齐横屏大厅动态：小飞机飞越、程序星光、少量糖果粒子和三个主按钮的轻量弹入/呼吸；不得改变按钮位置、点击逻辑和存档行为。
8. 为小游戏横屏大厅提供独立的飞机轨迹、粒子密度和安全区参数，由主开发会话写入 `minigame/game.js`，本会话负责截图或真机视觉验收。

整体美化会话不得直接修改：

- `minigame/game.js`
- `minigame/game.json`
- `minigame/project.config.json`
- `CHANGELOG.md`
- 版本号和发布记录

整体美化会话只允许在本轮修改 `web/index.html` 中与横屏大厅动效直接相关的 CSS/装饰层；不得修改网页玩法、棋盘、骰子、任务、存档和页面跳转。正式修改前仍须备份，并把文件交给主开发会话做发布前合并检查。

## 验收标准

1. 冷启动首先显示加载页，不出现黑屏或空白画面。
2. 进度百分比与资源完成数一致。
3. Logo 在微信真机显示正常。
4. 关键资源失败时可重试并显示失败信息。
5. 加载完成后自动进入大厅，按钮功能和存档读取不受影响。
6. 进入游戏后棋盘、棋子、任务卡和国王卡均显示正常。
7. 背景音乐开关、骰子音效、任务触发和棋子移动逻辑不回归。
8. 最终上传包小于 4MB。
9. 网页横屏 `1366×768`、`1440×900` 和极矮横屏 `844×390` 中，飞机与粒子不遮挡 Logo、三个主按钮、顶部 HUD 和底部功能入口。
10. 小游戏横屏至少检查 `844×390` 与 `932×430`，动态不得遮挡胶囊安全区或按钮触控区域。
11. 横竖屏均遵守低动态偏好或性能降级策略；页面隐藏后不继续进行无意义高频绘制。
