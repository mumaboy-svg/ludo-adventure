# 飞行棋大冒险 v2.x 视觉美化规划

> 目标：把《飞行棋大冒险》从“功能可用的网页游戏”逐步升级成“成熟小游戏质感”的界面。  
> 原则：先备份，再小步改；先网页预览确认，再决定是否同步 GitHub；每次网页端更新后都给“小程序口同步说明”。

## 事务性协作规则

为了避免主会话被重复性检查拖慢，后续按“Codex + Mavis + QClaw”协作推进。完整权限、测试门槛和任务包格式见 `docs/协作分工与测试准入.md`。

主会话负责：

- 确认视觉方向、页面优先级和最终审美判断。
- 修改核心代码、处理双端同步、执行本地验证。
- 决定是否进入 GitHub 发布流程。
- 输出每轮“小程序口同步说明”。

Mavis 负责事务性和可核对工作：

- 整理素材清单、图片尺寸、WebP 转换状态和缺图项。
- 对照 `web/` 与 `minigame/`，检查界面入口、素材路径、规则常量是否一致。
- 检查每轮是否已备份、是否漏写更新说明、是否有不该提交的文件。
- 根据主会话确定的方向，整理 AI 作图提示词初稿。
- 发布前做差异报告和测试清单核对。

QClaw 负责可自动化、结果明确的技术检查：

- 运行语法检查、链接和素材路径检查、版本一致性检查。
- 按固定视口执行布局截图和溢出扫描。
- 按既定测试用例做网页回归，并输出失败证据。
- 对比网页端和小程序端的常量、入口和素材差异。

Mavis 不负责：

- 自行改玩法规则、棋盘点位、骰子逻辑或存档结构。
- 自行决定最终 UI 风格。
- 未经同步包同时大改网页端和小程序端。

Mavis 和 QClaw 均不得直接发布、提交 Git、决定版本号或修改正式高风险模块。只有通过项目测试准入后，才可在隔离副本中承担低风险机械修复。

## 一、整体美术方向

推荐方向：**童趣冒险 + 桌游质感 + 轻微糖果立体风**。

关键词：

- 明亮、轻松、多人互动
- 像小游戏 App，不像后台工具
- 棋盘像实体桌游棋盘
- 任务卡、国王卡像真正抽出来的卡牌
- 按钮有立体感、点击反馈和发光层次
- 横版网页和竖版手机分别适配，不强行套一个布局

不建议：

- 不做过重的写实 3D 风，成本高且容易不统一。
- 不做太复杂的特效，避免影响手机端性能。
- 不在没有备份的情况下改骰子、棋盘规则、存档逻辑。

## 二、推进优先级总览

本规划把后续美化拆成三类：

1. **简单快速完成**：主要改 CSS / 少量 HTML，不依赖新图片，风险低，优先做。
2. **需要图片支持**：需要用户或 AI 生图提供素材，图片确认后再落地。
3. **大动作更新**：涉及页面结构、动画系统、骰子专项或双端同步，放后面分阶段做。

# A. 简单快速完成清单

这些适合近期一项一项推进，每次只做一个小模块。

## A1. 大厅信息层级精修（已完成：v2.7.8）

目标：大厅第一眼更像游戏首页。

内容：

- 主按钮“继续上局 / 新游戏 / 快速开始”保持最醒目。
- 次级入口“任务中心 / 设置 / 棋子测试 / 导入导出”降低视觉重量。
- 横版和竖版分别设定间距，保证手机第一屏完整显示。

风险：低。  
是否需要图片：不需要。  
是否影响玩法：不影响。

## A2. 右侧玩家进度卡片精修（已完成：v2.6.1）

目标：让游戏中右侧信息不再像普通列表。

内容：

- 玩家进度卡片加入当前玩家高亮。
- 玩家颜色标记更清晰。
- 每个玩家的进度条、终点状态、回合提示更像 HUD。
- 鼠标悬停或当前玩家时轻微发光。

风险：低。  
是否需要图片：不需要。  
是否影响玩法：不影响。

## A3. 游戏记录气泡化（已完成：v2.6.3）

目标：让“游戏记录”像游戏事件日志，而不是普通文本。

内容：

- 不同类型日志用不同小标签，例如：掷骰、移动、任务、国王卡、系统。
- 日志之间增加层次和间距。
- 长文字自动换行，不挤压布局。

风险：低。  
是否需要图片：不需要。  
是否影响玩法：不影响。

## A4. 弹窗基础统一（已完成：并入 v2.7.0）

目标：所有弹窗打开、关闭、按钮位置更统一。

内容：

- 统一关闭按钮位置。
- 统一弹窗出现动画。
- 统一确认/取消按钮样式。
- 不改任务内容和规则。

风险：低到中。  
是否需要图片：不需要。  
是否影响玩法：不影响。

## A5. 设置页基础美化（已完成：v2.7.5）

目标：设置页像游戏菜单，不像表单。

内容：

- 设置项分组：游戏设置、视觉设置、数据管理。
- 开关按钮做成滑动开关样式。
- 危险操作例如清空存档使用红色警告区。
- 按钮统一成游戏按钮风格。

风险：中低。  
是否需要图片：不需要。  
是否影响玩法：不影响，但要检查设置保存。

## A6. 任务中心基础美化（已完成：v2.7.6）

目标：任务中心变成任务卡册的感觉。

内容：

- 任务列表卡片化。
- 增加分类标签视觉，例如全部 / 大冒险 / 国王卡 / 自定义。
- 当前启用任务数更清楚。
- 导入导出按钮美化。

风险：中低。  
是否需要图片：不需要。  
是否影响玩法：不影响，但要检查导入导出。

## A7. 棋盘外框和舞台感（已完成：v2.7.7）

目标：不改变棋盘点位，只增强棋盘周围质感。

内容：

- 给棋盘外层加桌面底座、阴影、圆角或木质边框感。
- 棋盘与背景拉开层次。
- 当前玩家区域和棋盘之间有更自然的视觉连接。

风险：低。  
是否需要图片：不需要。  
是否影响玩法：不影响。

# B. 需要图片支持清单

这些需要先准备素材，图片确认后再转 WebP 并引用。

## B1. 大厅背景二次升级

需要图片：

- 横版大厅背景，建议 16:9，例如 1920×1080。
- 竖版大厅背景，建议 9:16，例如 1080×1920。

图片要求：

- 中央区域不要太花，方便放 Logo 和按钮。
- 四周可以有云朵、飞机、棋子、冒险地图、星星等元素。
- 色彩明亮，但不要和按钮颜色太接近。

落地内容：

- 替换大厅背景。
- 根据背景明暗调整按钮发光和文字描边。

## B2. 主按钮统一素材

需要图片：

- 继续上局按钮底图。
- 新游戏按钮底图。
- 快速开始按钮底图。
- 最好同一套风格、同一比例。

图片要求：

- 建议透明背景 PNG，之后转 WebP。
- 按钮不要自带太小的文字，避免缩放后糊。
- 如果图片包含文字，要保证横版和竖版都清晰。

落地内容：

- 统一主按钮比例。
- 解决不同分辨率变形问题。
- 增加按下状态和悬停光效。

## B3. 底部功能图标统一套装

需要图片：

- 任务中心
- 设置
- 棋子测试
- 导入导出
- 可选：返回大厅、音效开关、帮助说明

图片要求：

- 同一透视、同一光源、同一描边。
- 透明背景，适合放在不同背景上。
- 图标本身要比文字更醒目。

落地内容：

- 大厅底部入口统一成“大图标 + 短文字”。
- 加发光边缘，解决和背景混在一起的问题。

## B4. 任务卡 / 国王卡背景细化

已有素材：

- `国王卡.png`
- `任务卡.png`

后续可继续细化：

- 给普通任务、国王卡、奖励、惩罚做不同卡面。
- 卡面中间预留文字安全区。
- 边框不要压文字。

落地内容：

- 弹窗文字严格放入卡面安全区。
- 卡片弹出动画更像抽卡。

## B5. 骰子实体素材或骰子面素材

需要图片：

- 稳妥方案：1-6 六张骰子面图，统一立体高光。
- 更高级方案：一套骰子实体帧动画或 CSS 3D 辅助贴图。

图片要求：

- 每个点数必须清楚，尤其 2 和 5 不能混淆。
- 透视、阴影、颜色一致。
- 建议先做静态六面，再做动画。

落地内容：

- 骰子专项重做时使用。
- 先保证点数正确，再追求旋转效果。

## B6. 胜利结算图 / 奖励图

需要图片：

- 胜利标题牌
- 星星、彩带、奖杯、飞机冲线等装饰

落地内容：

- 胜利弹窗更有仪式感。
- 后续可用于分享图或小程序结算页。

## 三、Mavis 协作分工规则

参考写作项目里的协作方式：Codex 负责创意判断、界面设计、代码落地和最终把关；Mavis 可以承接非创造力、可检查、可整理、可重复的工作，减少主线被琐事打断。

### 1. Codex 负责的工作

Codex 负责需要审美判断、产品取舍、代码实现和最终决策的部分：

- 制定整体视觉方向和分阶段方案。
- 判断某个界面是否更像成熟小游戏。
- 设计大厅、主界面、任务中心、设置页、弹窗等核心交互。
- 修改网页代码、CSS、资源引用和逻辑。
- 判断是否影响骰子、棋盘、任务、存档等核心功能。
- 每轮更新后的最终说明、小程序同步说明、Git 提交建议。
- 出现问题时决定回退、修复或暂停。

### 2. Mavis 可分担的工作

Mavis 负责非创造力、可清单化、可复核的辅助工作：

- 整理素材清单：哪些图片已有、哪些缺失、哪些需要转 WebP。
- 检查图片命名、路径、尺寸、格式是否统一。
- 整理每轮更新记录，形成简短变更日志。
- 根据 Codex 的设计方向整理 AI 作图提示词初稿。
- 对比网页端和小程序端差异，列出需要同步的文件和模块。
- 根据截图整理问题清单，例如：遮挡、太近、文字溢出、按钮变形。
- 发布前按检查表逐项核对，不做主观设计决策。
- 整理 Git 状态摘要：哪些文件变了，哪些可能不该提交。

### 3. 不建议交给 Mavis 的工作

以下工作需要 Codex 或用户确认，不建议直接交给 Mavis 独立决定：

- 改游戏规则、棋盘点位、骰子逻辑、存档结构。
- 决定最终 UI 风格。
- 直接替换核心素材。
- 直接提交 GitHub 或发布线上版本。
- 对用户截图做最终审美判断。

### 4. 每轮推荐协作流程

```text
1. Codex 定义本轮目标，例如：大厅信息层级精修。
2. Mavis 整理辅助信息，例如：当前素材、截图问题、需同步项。
3. Codex 根据整理结果修改网页代码。
4. Mavis 按检查表复核：路径、WebP、溢出、同步说明、更新记录。
5. Codex 最终汇报给用户，并等待用户确认是否继续或推 GitHub。
```

### 5. Mavis 任务包模板

后续如果要把辅助工作分给 Mavis，可以用这个格式：

```text
Mavis 任务：XXX
范围：只检查/整理，不改核心代码。
输入：相关文件、截图或本轮目标。
输出：问题清单、素材清单、同步清单或提示词初稿。
限制：不改规则、不改骰子逻辑、不发布、不提交 GitHub。
```

## 五、每轮修改固定流程

每轮都按下面流程执行：

1. `git status -sb` 检查当前状态。
2. 在 `backups/` 创建本轮修改前备份。
3. 只修改本轮相关文件。
4. 本地打开 `http://127.0.0.1:8080/web/index.html` 检查。
5. 说明本次改了什么、没改什么。
6. 给“小程序口同步说明”。
7. 用户确认后再决定是否提交和推 GitHub。

## 六、小程序口同步说明模板

```text
小程序口同步说明：
本次网页端更新：XXX。
玩法规则是否变化：否 / 是。
小程序端是否需要同步：暂不需要 / 需要。
```

## 七、当前建议下一步

当前 A1-A7 快速美化项均已完成。建议下一步先做一次 **A 阶段双端整体验收**，确认大厅、游戏 HUD、任务中心、设置、弹窗和棋盘舞台在真机上的一致性，再进入需要新图片的 B 阶段。

原因：

- 汇总 A1-A7 的视觉一致性与移动端安全区问题。
- 先修复现有样式回归，再开始新素材替换。
- 不在整体验收中改骰子、棋盘规则或存档结构。
- B 阶段开始前再向用户确认首批图片需求和统一素材规范。
